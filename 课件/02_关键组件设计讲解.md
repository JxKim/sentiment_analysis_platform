# 尚舆分析平台 —— 关键组件设计讲解

---
在实际开发过程中，在具体编码前，我们需要先对项目的整体架构进行设计，包括项目的模块划分、组件设计、数据库设计、接口设计等。

## 1. 多智能体架构设计

### 1.1 多智能体项目架构设计概述

在现代 AI 应用开发中，单个 LLM 调用往往无法满足复杂任务的需求。多智能体（Multi-Agent）架构将复杂问题拆解为多个子任务，由不同的 Agent 分别处理，再汇总结果。在选择架构方案之前，我们需要理解几种主流的设计模式。

#### 1.1.1 并行编排（Parallel Orchestration）

多个 Agent 同时对一个任务的不同维度进行分析，各自独立运行，最后汇总结果。

```
用户查询: "分析某品牌最近一周的舆情状况"
          │
          ├── Agent A: 社交媒体舆情分析
          │    └── 搜索微博、知乎、小红书等平台 → 情感分析 + 关键词提取
          │
          ├── Agent B: 媒体报道分析
          │    └── 搜索新闻网站、行业媒体 → 传播路径 + 媒体立场分析
          │
          └── Agent C: 信息真实性核验
               └── 搜索官方数据、权威来源 → 数据交叉验证 + 来源可信度评级
                        │
                        ▼
                  汇总 → 综合分析报告
```

- **优点**：速度快，各 Agent 无依赖关系，可最大化并行度
- **缺点**：Agent 之间缺乏信息交流，可能遗漏跨维度的洞察
- **适用场景**：多维度独立分析，各维度之间不需要相互参照

#### 1.1.2 顺序编排（Sequential Pipeline）

Agent 按照预定顺序依次执行，前一个 Agent 的输出作为后一个 Agent 的输入。

```
用户查询 → Agent A(初步搜索) → Agent B(深度分析) → Agent C(生成报告)
```

- **优点**：流程清晰，每步可以充分利用前序结果
- **缺点**：延迟高，并行度低，上游错误会级联放大
- **适用场景**：数据需要逐步加工的场景，如"搜索 → 筛选 → 分析 → 报告"

#### 1.1.3 层级路由（Hierarchical / Router）

一个主控 Agent（Orchestrator）接收用户意图，根据意图类型将任务路由到不同的专业 Agent。

```
用户查询 → Orchestrator Agent
              │
              ├── 舆情分析类 → InsightAgent
              ├── 事实查询类 → QueryAgent
              ├── 对比分析类 → 同时调度 InsightAgent + MediaAgent
              └── 报告生成类 → ReportAgent
```

- **优点**：灵活性高，可根据任务类型动态选择策略
- **缺点**：Orchestrator 本身需要较强的意图理解能力，增加了设计复杂度
- **适用场景**：需要处理多种不同类型任务的系统

#### 1.1.4 协作讨论（Collaborative / Roundtable）

多个 Agent 围绕同一主题进行多轮讨论，类似专家圆桌会议。每个 Agent 可以在其他 Agent 发言的基础上补充、质疑或深化分析。

```
             ┌──────────────┐
             │  共享上下文   │  ← 各 Agent 的发言对所有人可见
             └──────┬───────┘
        ┌───────────┼───────────┐
        ▼           ▼           ▼
   ┌─────────┐ ┌─────────┐ ┌─────────┐
   │Agent A  │ │Agent B  │ │Agent C  │
   │(Insight)│ │(Media)  │ │(Query)  │
   └────┬────┘ └────┬────┘ └────┬────┘
        │           │           │
        └───────────┼───────────┘
                    ▼
           ┌─────────────┐
           │  Host Agent │  ← 主持人总结/引导
           └─────────────┘
```

- **优点**：信息流动充分，可以通过多轮讨论产生更深刻的洞察
- **缺点**：Token 消耗大，需要有效的讨论管理机制（避免跑题、冗余、冲突）
- **适用场景**：需要多视角深度研判的复杂分析任务

#### 1.1.5 Map-Reduce 模式

将大量数据分片，每个 Agent 处理一个分片（Map），最后由聚合 Agent 汇总结果（Reduce）。

```
大数据集 → [分片1 → Agent1] ┐
          [分片2 → Agent2] ├→ Reduce Agent → 汇总结果
          [分片3 → Agent3] ┘
```

- **优点**：适合海量数据处理场景
- **缺点**：分片策略和汇总逻辑需要精心设计
- **适用场景**：海量文档分析、大规模数据挖掘

#### 1.1.6 各架构模式对比

| 架构模式 | 并行度 | Agent 间交互 | 延迟 | 复杂度 | 适用场景 |
|---------|--------|-------------|------|--------|---------|
| 并行编排 | 高 | 无 | 低 | 低 | 多维度独立分析 |
| 顺序编排 | 低 | 单向传递 | 高 | 低 | 流水线式加工 |
| 层级路由 | 中 | 单向调度 | 中 | 中 | 多类型任务分发 |
| 协作讨论 | 高 | 多轮交互 | 高 | 高 | 深度研判分析 |
| Map-Reduce | 高 | 仅汇总 | 中 | 中 | 海量数据并行处理 |

### 1.2 本项目中多智能体架构设计

本项目没有选择单一的架构模式，而是根据舆情分析的业务特点，设计了一种**混合架构**，融合了"并行编排"和"协作讨论"两种模式的优点。

#### 1.2.1 为什么选择混合架构？

舆情分析任务有三个关键需求，单一模式无法同时满足：

| 需求 | 单一模式的局限 |
|------|---------------|
| **覆盖面要广**（社交媒体 + 新闻报道 + 权威数据） | 纯顺序编排太慢，无法在合理时间内完成 |
| **分析要有深度**（不同视角的交叉印证） | 纯并行编排各自为战，缺少跨维度洞察 |
| **结果要结构化**（最终产出格式化报告） | 并行和讨论模式的输出是非结构化的 |

因此我们设计了三阶段流水线：

```
         第一阶段                  第二阶段                 第三阶段
      (并行编排)               (协作讨论)              (顺序汇总)

   ┌──────────┐            ┌──────────────┐         ┌──────────────┐
   │ Insight  │──────┐     │              │         │              │
   │ Engine   │      │     │  ForumEngine │         │ ReportEngine │
   │(本地DB)  │      │     │              │         │              │
   └──────────┘      │     │  监听各引擎   │         │  汇总三引擎   │
                     ├────►│  中间结果     ├────────►│  报告+论坛    │
   ┌──────────┐      │     │              │         │  日志         │
   │  Media   │──────┤     │  HOST主持人   │         │              │
   │ Engine   │      │     │  定期总结     │         │  生成HTML/    │
   │(网络搜索)│      │     │              │         │  MD/PDF报告   │
   └──────────┘      │     └──────┬───────┘         └──────────────┘
                     │            │
   ┌──────────┐      │     HOST发言被缓存
   │  Query   │──────┘     给各引擎
   │ Engine   │       (摘要/反思摘要时参考)
   │(权威核实)│
   └──────────┘
```

#### 1.2.2 第一阶段：并行研究

三个研究引擎在同一时刻被启动，各自独立运行。设计要点：

**引擎角色分化**：每个引擎有不同的搜索范围和分析视角，而不是简单的"三个引擎做同样的事"。

| 引擎 | 数据来源 | 核心能力 | 分析视角 |
|------|---------|---------|---------|
| **InsightEngine** | 本地 MySQL（微博/知乎/B站/小红书等） | 关键词优化、情感分析、聚类去重 | 公众情绪和舆论倾向 |
| **MediaEngine** | 网络搜索（Bocha/Tavily/Anspire） | 综合搜索、时效搜索、多模态搜索 | 媒体报道和传播路径 |
| **QueryEngine** | Tavily 搜索 + 权威域增强 | 来源可信度分级、官方域名补充搜索 | 官方信息和数据核验 |

**为什么要区分角色？**

如果三个引擎都是相同的搜索逻辑，并行就没有意义——它们的输出会高度重叠。通过角色分化，每个引擎从一个独特的"专业视角"切入同一主题，产出互补的分析结果。这模拟了现实中的舆情分析团队分工：有人看社交媒体，有人看媒体报道，有人查官方数据。

**并行策略选择**：

使用 `threading.Thread` 而非 `asyncio`，原因：
- 引擎内部是同步的 LangGraph 工作流，不是 IO 密集型
- LLM API 调用虽然涉及网络等待，但使用同步 SDK（OpenAI SDK）
- 三个线程足以覆盖并行需求，不需要更复杂的协程调度

#### 1.2.3 第二阶段：协作讨论

这是本架构最独特的设计。第一阶段三个引擎独立运行，但它们之间存在信息孤岛的问题——InsightEngine 可能不知道 MediaEngine 发现了什么重要信息，反之亦然。

**设计思路**：引入一个 ForumEngine 作为"虚拟主持人"，监听各引擎的研究进展，定期组织讨论。

```
研究引擎A ████████████████████████████████████████████████ 完成
              │                    │               │
              ▼                    ▼               ▼
           SUMMARY_READY      SUMMARY_READY    SUMMARY_READY
          (段落摘要就绪)      (段落摘要就绪)    (段落摘要就绪)
              │                    │               │
              └────────────────────┼───────────────┘
                                  ▼
                    ┌─────────────────────────┐
                    │     ForumEngine          │
                    │                         │
                    │  _buffer = [摘要1,       │
                    │            摘要2,        │
                    │            摘要3,        │
                    │            摘要4,        │
                    │            摘要5]  ← 攒够了! │
                    │                         │
                    │  触发 HOST 发言:          │
                    │  "Insight 从社交媒体发现 │
                    │   用户情绪偏向负面...      │
                    │   Media 报道集中在A角度... │
                    │   Query 找到官方数据与    │
                    │   媒体报道存在矛盾...       │
                    │   建议下一步重点核实..."   │
                    └────────────┬────────────┘
                                 │
                    HOST发言被缓存，各引擎在
                    摘要/反思摘要时读取
                                 │
              ┌──────────────────┼──────────────────┐
              ▼                  ▼                  ▼
        InsightEngine     MediaEngine        QueryEngine
        (摘要更新：       (摘要更新：         (摘要更新：
         参考HOST建议，     参考HOST建议，      参考HOST建议，
         修正分析角度)      修正分析角度)        修正分析角度)
```

**关键设计决策**：

1. **触发频率**：每收集 5 条摘要触发一次 HOST 发言。频率太低则讨论不够及时，太高则 Token 消耗过大且发言可能空洞。

2. **HOST 发言的时效性**：各引擎读取的是"最新一条" HOST 发言，而不是历史发言。这是有意设计的——每轮 HOST 发言已综合前面所有讨论，读最新一条即可获取完整的"讨论进展"。

3. **HOST 的角色定位**：HOST 不产生新的事实信息，而是做"观点整合 + 矛盾识别 + 方向建议"。这确保了它不会引入幻觉内容。

4. **去中心化**：ForumEngine 不控制其他引擎，只提供参考意见。各引擎自行决定是否采纳 HOST 建议。这种松耦合保证了容错性——ForumEngine 出问题不会阻塞研究流程。

#### 1.2.4 第三阶段：顺序汇总

前两个阶段产出的是"原材料"——三份 Markdown 报告和一份论坛讨论日志。第三阶段的 ReportEngine 将这些原材料加工为结构化报告。

这一阶段采用顺序流水线模式，因为报告生成天然是顺序的：先选模板 → 再定结构 → 再分配篇幅 → 再逐章生成 → 最后组装渲染。每一步依赖前一步的输出。

#### 1.2.5 架构设计总结

| 设计决策 | 选择 | 核心理由 |
|---------|------|---------|
| 为什么第一阶段用并行 | 三个引擎数据源不同，无依赖 | 速度最大化 |
| 为什么第二阶段引入讨论 | 跨维度洞察需要信息交换 | 弥补并行模式的"信息孤岛" |
| 为什么第三阶段用顺序 | 报告生成步骤有先后依赖 | 流程自然匹配 |
| 为什么用 EventBus 而非直接调用 | 引擎和 Forum 应解耦 | 容错 + 可扩展 |
| 为什么 HOST 发言是"建议"而非"指令" | 避免集中控制，保持引擎自主性 | 松耦合 |

## 2. 单个研究引擎节点流程设计

三个研究引擎（InsightEngine、MediaEngine、QueryEngine）共享相同的内部流程架构。本节以 InsightEngine 为代表讲解单个引擎的内部设计。

### 2.1 为什么选择 LangGraph？

在单引擎内部，我们的核心需求是：**定义一个多步骤的、可迭代的、有状态的研究工作流**。

| 需求 | LangGraph 的对应能力 |
|------|-------------------|
| 多步骤流程 | 用节点（Node）和边（Edge）定义 DAG 执行图 |
| 条件分支 | 条件边（Conditional Edge），根据运行时状态决定下一步 |
| 循环迭代 | 支持循环边，实现"反思 → 再搜索"的迭代模式 |
| 状态传递 | TypedDict State 在工作流节点间自动传递和更新 |
| 可观测性 | 每个节点的输入输出可通过 LangGraph 内置机制追踪 |

备选方案对比：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 手写 if/else 流程 | 灵活 | 复杂流程难以维护，状态管理混乱 |
| CrewAI / AutoGen | 开箱即用 | 黑盒度高，难以定制节点行为 |
| LangGraph | 显式图定义，状态可控，社区活跃 | 学习曲线稍高 |

### 2.2 研究工作流设计

研究过程被建模为一个**带有反思循环的段落迭代流程**：

```
                        ┌─────────────────────┐
                        │  ① 生成报告大纲      │
                        │  (5个段落,每段一个   │
                        │   分析维度)          │
                        └──────────┬──────────┘
                                   │
                        ┌──────────▼──────────┐
                        │                     │
                        │   段落迭代循环       │
                        │   (遍历每个段落)      │
                        │                     │
                        │  ┌───────────────┐  │
                        │  │ ② 初次搜索     │  │  ← LLM生成搜索关键词
                        │  │   执行搜索     │  │  ← 调用搜索工具
                        │  └───────┬───────┘  │
                        │          │          │
                        │  ┌───────▼───────┐  │
                        │  │ ③ 初次摘要     │  │  ← 基于搜索生成分析
                        │  │   发布SUMMARY  │  │  ← 触发Forum讨论
                        │  │   _READY事件   │  │
                        │  └───────┬───────┘  │
                        │          │          │
                        │  ┌───────▼───────┐  │
                        │  │ 反思循环       │  │
                        │  │ (最多N次)      │  │
                        │  │               │  │
                        │  │ ④ 反思搜索    │  │  ← 分析不足→搜什么
                        │  │ ⑤ 反思摘要    │  │  ← 融合新结果→更新
                        │  │               │  │
                        │  │ 还有改进空间? │  │
                        │  │ ──是──→ 继续   │  │
                        │  │ ──否──→ 退出   │  │
                        │  └───────┬───────┘  │
                        │          │          │
                        │  还有更多段落?       │
                        │  ──是──→ 回到②      │
                        │  ──否──→ 退出循环    │
                        └──────────┬──────────┘
                                   │
                        ┌──────────▼──────────┐
                        │  ⑥ 格式化最终报告    │
                        │  (汇总所有段落摘要)  │
                        └──────────┬──────────┘
                                   │
                        ┌──────────▼──────────┐
                        │  ⑦ 保存报告到磁盘    │
                        │  (.md + .json)      │
                        └─────────────────────┘
```

### 2.3 核心设计：反思循环

反思循环是本工作流最核心的设计理念——让 Agent 具备"审视自己的不足并主动补充"的能力。

**为什么需要反思？**

一次搜索往往不够深入，原因包括：
- 首次搜索的关键词可能遗漏重要角度
- 搜索结果的信息量可能不足
- 不同来源的信息可能存在矛盾需要核实
- 初次分析可能过于表面

**反思循环的工作机制**：

```
当前摘要 ──→ LLM 审视 ──→ 发现不足 ──→ 生成新搜索词 ──→ 执行补充搜索
                                                              │
                                                              ▼
                    ┌─── 更新摘要 ←── 融合新旧结果 ←── 获得补充信息
                    │
                    │  (更新后是否还有改进空间?)
                    │       │            │
                    │      是            否
                    │      │            │
                    └── 再来一轮       退出反思
```

**关键设计决策**：

1. **反思次数上限**：默认每段落最多 2 轮反思。过多的反思轮次会导致 Token 消耗急剧增加，且边际收益递减。

2. **反思 vs 重新搜索**：反思不是简单的"再搜一次"，而是"分析当前摘要的不足 → 针对性地搜索补充信息"。这需要 LLM 在反思搜索节点中同时完成"缺口分析"和"搜索计划制定"两个认知任务。

3. **讨论对反思的影响**：在摘要和反思摘要阶段，各引擎会读取最新的 HOST 发言。如果 HOST 指出了某个引擎忽略的角度，该引擎会在更新段落分析时参考这个方向；当前实现不是在搜索节点直接读取 HOST 发言。

4. **段落间的信息传递**：当引擎从段落 0 进入段落 1 时，前一段落的搜索结果和摘要不会丢失（保存在 State 中），后一段落的搜索可以参照前面的发现。

### 2.4 工作流状态（State）设计

状态对象是工作流的"记忆"，在所有节点间流转、被各节点读取和更新。

| 状态域 | 字段 | 说明 |
|--------|------|------|
| **输入** | `query` | 用户的原始查询 |
| **大纲** | `report_title`, `paragraphs` | LLM 生成的报告标题和 5 段式大纲 |
| **迭代控制** | `current_paragraph_index` | 当前在处理第几个段落 (0~4) |
| | `current_reflection_count` | 当前段落的反思次数 |
| | `max_reflections` | 每段落最多反思次数（默认 2） |
| **搜索结果** | `paragraphs[i].research.search_history` | 当前段落所有搜索的完整记录（关键词、工具、结果） |
| **分析结果** | `paragraphs[i].research.latest_summary` | 当前段落最新版摘要，每次反思后更新 |
| **最终输出** | `final_report`, `is_completed`, `error` | 最终报告文本、完成标志、错误信息 |

**状态设计原则**：

- **最小化**：只存节点间需要传递的数据，不存中间计算产物
- **可序列化**：所有状态字段都可以被序列化为 JSON，便于调试和持久化
- **增量更新**：每个节点只返回自己改变的字段，不重写整个 State

### 2.5 三个引擎的差异化设计

虽然流程一致，但三个引擎在设计层面有关键差异：

| 设计维度 | InsightEngine | MediaEngine | QueryEngine |
|---------|-------------|------------|------------|
| **数据源** | 本地 MySQL 社交媒体数据 | Bocha/Tavily 网络搜索 | Tavily 搜索 + 权威域增强查询 |
| **搜索工具数** | 5（热榜/全局/日期/评论/平台） | 5（综合/纯Web/结构化/24h/一周） | 5（综合/纯Web/结构化/24h/一周） |
| **特有处理层** | 关键词优化 + 情感分析 + 聚类去重 | 多模态/网页搜索适配 | 来源可信度分级 + 官方域名补充搜索 |
| **LLM 角色设定** | "公众舆情分析师" | "媒体传播分析师" | "信息核验专家" |
| **搜索语言风格** | 口语化、网络用语 | 正式媒体用语 | 官方术语、专业词汇 |
| **Prompt 侧重点** | 真实公众情绪、避免官话套话 | 传播路径、媒体立场、报道角度 | 数据真实性、交叉验证、来源权威性 |
| **输出侧重** | 舆论倾向、情感分布、热点话题 | 报道数量、媒体层级、传播链 | 官方信息梳理、数据一致性、可信度评估 |

这些差异不是代码层面的简单配置，而是从 **Prompt 设计、工具选择、处理流程** 三个层面系统性地体现。

### 2.6 研究流程中的 Forum 协作点

引擎内部有两个节点承担了与 ForumEngine 的交互：

| 节点 | 交互方式 | 交互内容 |
|------|---------|---------|
| **初次摘要** | 发布 `SUMMARY_READY` 事件 | 将刚完成的段落摘要广播给 ForumEngine |
| | 读取 `get_latest_host_speech()` | 获取 HOST 最新的讨论总结 |
| **反思摘要** | 读取 `get_latest_host_speech()` | 在更新段落分析时参考 HOST 建议 |

第一次 Forum 交互发生在初次摘要完成之后。第二次开始，各引擎就能收到 HOST 的反馈意见，从而在后续摘要更新和下一个段落分析中调整方向。

## 3. 报告生成流程设计

ReportEngine 负责将三个引擎的研究报告和论坛讨论记录汇总为结构化的 HTML 报告。这是整个系统中**节点最多、流程最长**的引擎。

### 3.1 总体流程

```
输入: Insight报告.md + Media报告.md + Query报告.md + 论坛日志.log
  │
  ▼
① 输入标准化          将四份输入文件转为统一的 dict 结构
  │
  ▼
② 模板选择            LLM 根据查询内容从 6 种模板中选择最适合的
  │
  ▼
③ 模板切分            将模板 Markdown 解析为章节列表
  │
  ▼
④ 版式设计            LLM 设计标题/副标题/导语/主题配色
  │
  ▼
⑤ 字数预算            为每章分配目标/最小/最大字数（总计约 4 万字）
  │
  ▼
⑥ 上下文装配          将模板+版式+预算+源报告组装为章节生成的输入
  │
  ▼
⑦ 逐章生成            每个章节独立调用 LLM，生成结构化 JSON
  │   ├── JSON 解析 + 多引擎修复
  │   └── 结构验证 + 内容净化
  │
  ▼
⑧ 文档缝合            将所有章节 JSON 合并为完整文档
  │
  ▼
⑨ HTML 渲染           将文档 JSON 渲染为交互式 HTML
  │
  ▼
⑩ 报告保存            持久化 HTML + JSON + 状态文件
  │
  ▼
输出: HTML报告 + 中间表示JSON + 状态JSON
```

### 3.2 核心设计：中间表示（IR）

IR（Intermediate Representation，中间表示）是本引擎最重要的设计决策。

**问题背景**：

在多源报告汇总时，有两个自然的方案：

| 方案 | 做法 | 问题 |
|------|------|------|
| A: LLM 直接生成 HTML | 把引擎报告扔给 LLM，让它生成 HTML 页面 | HTML 质量不稳定，样式错误、结构混乱、库引用错误频发 |
| B: LLM 生成 Markdown | LLM 输出 Markdown，前端渲染 | Markdown 表达能力有限，缺少图表、KPI 卡片、SWOT 表等专业元素 |

**IR 方案**：

在 LLM 输出和最终渲染之间加入一个结构化的中间层：

```
LLM 生成文本                  IR JSON                      最终输出
┌────────────┐         ┌─────────────────┐         ┌──────────────┐
│            │         │ {               │         │              │
│  章节生成   │────────►│   "blocks": [   │────────►│  HTML 报告    │
│  Prompt     │         │     {heading},  │         │  (Chart.js + │
│            │         │     {paragraph}, │         │   MathJax)   │
│            │         │     {table},     │         │              │
│            │         │     {chart},     │────────►│  Markdown    │
│            │         │     {kpiGrid},   │         │  (图表降级)   │
│            │         │     ...          │         │              │
│            │         │   ]             │────────►│  PDF         │
│            │         │ }               │         │  (WeasyPrint)│
└────────────┘         └─────────────────┘         └──────────────┘

                     LLM 只需理解 JSON Schema
                     渲染交给专门的 Renderer
                     一份 IR → 多种输出格式
```

**IR 的设计原则**：

1. **表达力足够**：覆盖舆情报告常用元素，共 16 种块类型
2. **约束明确**：每种块类型的字段有严格 Schema，减少 LLM 自由发挥空间
3. **与渲染解耦**：IR 不含样式信息，渲染逻辑完全在 Renderer 中

**16 种内容块类型**：

| 类别 | 块类型 | 用途 |
|------|--------|------|
| 基础文本 | `heading` | 多级标题 |
| | `paragraph` | 正文段落，支持内联标记 |
| | `blockquote` | 引用块 |
| | `list` | 有序/无序列表 |
| | `hr` | 分割线 |
| | `code` | 代码块 |
| 数据可视化 | `table` | 常规表格 |
| | `widget` | 图表（柱状/折线/饼图/雷达/散点等） |
| 专业分析 | `swotTable` | SWOT 分析矩阵 |
| | `pestTable` | PEST 分析表 |
| | `kpiGrid` | KPI 指标卡片组 |
| 引擎特有 | `engineQuote` | 直接引用某引擎的分析原文 |
| 布局装饰 | `callout` | 提示/警告/信息框 |
| | `figure` | 图片 |
| | `toc` | 目录占位符 |
| | `math` | 数学公式 |

### 3.3 模板系统设计

ReportEngine 内置 6 种报告模板，每种针对不同分析场景预设了章节结构。

| 模板名称 | 主题特点 | 典型章节 |
|---------|---------|---------|
| **企业品牌分析** | 品牌声量、美誉度、用户画像 | 品牌概况 → 声量趋势 → 竞品对比 → 品牌 SWOT → 建议 |
| **市场竞争分析** | 行业格局、市场份额、竞品动态 | 市场概况 → 竞争格局 → 竞品分析 → 市场 SWOT → 战略建议 |
| **日常监测报告** | 周期性舆情巡测 | 监测概览 → 平台分布 → 情感分析 → 热点话题 → 风险预警 |
| **政策行业分析** | 政策影响、行业趋势 | 政策解读 → 行业影响 → PEST 分析 → 机遇挑战 → 应对建议 |
| **社会热点分析** | 突发事件、社会舆论 | 事件概述 → 传播脉络 → 舆论分化 → 关键节点 → 趋势研判 |
| **突发事件分析** | 危机应对、传播链 | 事件概述 → 时间线 → 传播链 → 风险矩阵 → 应对建议 |

**模板选择机制**：

LLM 根据用户查询的内容特征自动选择模板，判断依据包括：
- 查询是否提到企业/品牌名称 → 企业品牌分析
- 查询是否提到"竞争/对比/市场份额" → 市场竞争分析
- 查询是否涉及政策/法规/行业 → 政策行业分析
- 查询是否关于突发事件/危机 → 突发事件分析
- 除此之外的通用查询 → 社会热点分析

### 3.4 章节生成的容错设计

这是整个流水线中**最易出错的环节**——LLM 需要生成复杂的嵌套 JSON 结构，且格式必须严格符合 IR Schema。实际测试中 LLM 生成 JSON 的错误率相当高。

为此设计了一套**四层防护机制**：

```
第一层：LLM 直接生成
  │  使用严格的 JSON Schema 约束 Prompt
  │  预期成功率：60-70%
  ▼ (失败)
第二层：本地自动修复
  │  - 清理 Markdown 代码块包装 (```json ... ```)
  │  - 修复括号配对不完整
  │  - 移除尾部多余逗号
  │  - 转义字符串中的控制字符
  │  - 使用 json_repair 库自动修复
  │  累计成功率：85-90%
  ▼ (仍然失败)
第三层：跨引擎 LLM 修复
  │  将损坏的 JSON 发送给其他引擎的 LLM 修复
  │  尝试顺序：ReportEngine → ForumEngine → InsightEngine → MediaEngine
  │  不同 LLM 对 JSON 修复的能力有差异，多引擎增加成功概率
  │  累计成功率：95-98%
  ▼ (全部失败)
第四层：占位章节
  │  使用预设的默认结构作为占位内容
  │  标记该章节为"生成失败"，但不中断整体流程
  │  保证 100% 有输出，哪怕是降级版本
```

**为什么需要跨引擎修复？**

因为不同 LLM 对 JSON 的"理解"不同。一个 LLM 生成的损坏 JSON，可能另一个 LLM 反而更容易修复。实践中发现，用同一个 LLM 修复自己生成的损坏 JSON 往往重复同样的错误模式，切换 LLM 可以打破这种模式。

### 3.5 内容净化（Sanitization）

即使 JSON 解析成功，LLM 生成的内容也可能违反 IR Schema 的结构约束。净化层解决这个问题：

| 净化项 | 常见 LLM 错误 | 处理方式 |
|--------|-------------|---------|
| 表格嵌套 | 在表格单元格内又嵌套列表/段落 | 递归展开为纯文本 |
| 空块 | 生成空的 blocks 数组或 None 值 | 替换为默认占位段落 |
| engineQuote 约束 | 在引擎引用中使用禁止的块类型（如 list、math） | 移除违规块，降级为 paragraph |
| SWOT 字段 | impact 字段使用了非法的评级值 | 映射到最近的合法值 |
| 图表数据 | 图表数据中的字符串/数值类型错误 | 强制类型转换 |

### 3.6 渲染器设计

得益于 IR 的设计，渲染层完全独立于生成层。同一份 IR JSON 可以产生三种输出：

| 渲染器 | 目标格式 | 核心依赖 | 特点 |
|--------|---------|---------|------|
| HTMLRenderer | 单文件 HTML | Chart.js, MathJax | 交互式图表、暗色模式、可导出 |
| MarkdownRenderer | Markdown | 无 | 图表降级为表格，适合文本处理 |
| PDFRenderer | PDF | WeasyPrint, Pango/Cairo | 图表转 SVG 后渲染，适合打印/分发 |

**HTML 渲染器**输出的报告是自包含的单文件，这意味着：
- 所有 CSS/JS 内嵌在 HTML 中，不需要外部文件
- 可以直接在浏览器中打开，不需要服务器
- 支持通过浏览器打印为 PDF

**PDF 渲染器**的预处理步骤：
1. 将 Chart.js 图表转为 matplotlib SVG
2. 将 LaTeX 公式转为 mathtext SVG
3. 将词云转为 PNG 图片
4. 将以上 SVG/PNG 嵌入 HTML
5. 用 WeasyPrint 将 HTML 渲染为 PDF

## 4. 通信机制设计

通信是分布式系统的核心问题。本项目中的通信需求可以归纳为两个问题：
1. 模块之间如何交换信息（引擎间通信）？
2. 后端如何将运行状态实时推送给前端（前后端通信）？

### 4.1 通信需求分析

| 通信场景 | 参与者 | 方向 | 特点 |
|---------|--------|------|------|
| 研究进度广播 | 各引擎 → 所有监听者 | 一对多 | 实时、低延迟 |
| 论坛摘要收集 | 各引擎 → ForumEngine | 多对一 | 异步、可缓冲 |
| HOST 发言反馈 | ForumEngine → 各引擎 | 一对多 | 非实时、拉取模式 |
| 进度推送到前端 | 引擎线程 → 浏览器 | 多对多 | 实时、需支持断线重连 |
| 错误通知 | 各引擎 → 前端 | 多对一 | 实时、不可丢失 |

这些场景中，有的是**一对多广播**（进度），有的是**多对一收集**（论坛摘要），有的是**拉取模式**（HOST 发言）。如果每个场景都单独设计通信通道，系统会变得极其复杂。

### 4.2 解决方案：EventBus 发布-订阅模式

我们统一使用**发布-订阅（Pub-Sub）模式**作为项目内部通信的基础设施。

#### 4.2.1 设计理念

```
                     ┌─────────────────────────┐
                     │        EventBus          │
                     │                          │
                     │  subscribe(callback)     │
                     │  unsubscribe(callback)   │
                     │  publish(event, data)    │
                     │                          │
                     │  _subscribers: [...]     │
                     └────────────┬─────────────┘
                                  │
              ┌───────────────────┼───────────────────┐
              │                   │                   │
              ▼                   ▼                   ▼
        订阅者 A             订阅者 B             订阅者 C
     (SSE 路由器)         (ForumEngine)        (日志记录器)
```

发布者**不知道**谁在订阅，订阅者**不知道**谁在发布。双方只依赖 EventBus 这个抽象中间层。

#### 4.2.2 事件类型体系

所有通信都通过统一的事件类型进行：

| 事件类型 | 发布者 | 订阅者 | 含义 |
|---------|--------|--------|------|
| `SUMMARY_READY` | Insight/Media/Query 引擎 | ForumEngine | 段落摘要就绪 |
| `ENGINE_PROGRESS` | Insight/Media/Query 引擎 | SSE 路由器 → 前端 | 引擎进度更新 |
| `ENGINE_RESULT` | Insight/Media/Query 引擎 | SSE 路由器 → 前端 | 引擎执行完成 |
| `ENGINE_ERROR` | Insight/Media/Query 引擎 | SSE 路由器 → 前端 | 引擎执行错误 |
| `FORUM_MESSAGE` | ForumEngine | forum_reader + SSE → 前端 | 论坛新消息 |
| `CONSOLE_OUTPUT` | 各模块 | SSE 路由器 → 前端 | 控制台日志 |

#### 4.2.3 关键设计考量

**线程安全**：

发布者可能在后台线程（搜索引擎在 `threading.Thread` 中运行），订阅者可能在主线程（SSE 路由器在 asyncio 事件循环中）。EventBus 使用 `threading.Lock` 保护订阅者列表，保证跨线程操作的安全性。

**错误隔离**：

一个订阅者的异常不应该影响其他订阅者或发布者。EventBus 在遍历订阅者时，每个回调都被 try/except 包裹，异常被静默捕获。

**防死锁设计**：

如果订阅者 A 的回调中调用了 `unsubscribe()` 移除自己，而发布者正持有锁遍历订阅者列表，就会出现死锁。解决方案是：遍历前先复制订阅者列表并释放锁，然后遍历副本执行回调。

```
正确的做法：
  1. 获取锁
  2. 复制 _subscribers → snapshot
  3. 释放锁
  4. 遍历 snapshot 执行回调  ← 此时回调中 unsubscribe 不会有问题

错误的做法：
  1. 获取锁
  2. 遍历 _subscribers 执行回调  ← 回调中 unsubscribe 需要获取同一个锁 → 死锁
```

### 4.3 引擎间通信线路（Insight/Media/Query ↔ ForumEngine）

这是本系统的**核心通信回路**，实现了引擎间的"协作讨论"。

```
                   EventBus (SUMMARY_READY)
                          │
        ┌─────────────────┼─────────────────┐
        │ publish          │                 │ publish
        ▼                  │                 ▼
   ┌─────────┐             │            ┌─────────┐
   │Insight  │             │            │  Media  │
   │Engine   │             │            │ Engine  │
   └─────────┘             │            └─────────┘
                           │
                    ┌──────┴──────┐
                    │ subscribe   │
                    ▼             │
              ┌──────────┐       │
              │  Forum   │       │
              │  Engine  │       │
              │          │       │
              │ 缓冲摘要  │       │
              │ 攒满5条→  │       │
              │ 触发HOST  │       │
              └────┬─────┘       │
                   │             │
          FORUM_MESSAGE          │
              事件               │
                   │             │
                   ▼             │
            ┌─────────────┐     │
            │forum_reader │     │
            │缓存最新HOST  │     │
            └──────┬──────┘     │
                   │             │
      get_latest_  │             │
      host_speech  │             │
                   ▼             ▼
              ┌─────────┐  ┌─────────┐
              │Insight  │  │  Media  │   ← 各引擎在摘要更新时读取
              │摘要更新 │  │摘要更新 │
              └─────────┘  └─────────┘
```

**设计要点**：

1. **缓冲触发而非即时触发**：ForumEngine 不会每条摘要都触发 HOST 发言，而是攒满 5 条再触发。这保证了 HOST 发言有足够的"素材"，避免发言空洞。

2. **拉取而非推送**：HOST 发言不主动推送给各引擎，而是由各引擎在需要时主动拉取（`get_latest_host_speech()`）。这是为了让各引擎保持在"自己决定何时参考 HOST 意见"的自主性。

3. **只保留最新发言**：forum_reader 只缓存最新一条 HOST 发言。因为 HOST 每轮发言已综合前面所有讨论，最新一条就是最全面的。

### 4.4 前后端通信线路（Engine → SSE → Browser）

这条线路解决"用户如何实时看到后台进度"的问题。

#### 4.4.1 为什么选择 SSE 而非 WebSocket？

| 对比维度 | SSE | WebSocket |
|---------|-----|-----------|
| 通信方向 | 单向：服务器 → 客户端 | 双向 |
| 协议 | HTTP，无需升级 | 需要 HTTP Upgrade |
| 代理兼容性 | 原生 HTTP，所有代理都支持 | 部分代理不支持 |
| 自动重连 | 浏览器原生支持 | 需手动实现 |
| 消息格式 | 文本，简单 | 二进制/文本，复杂 |
| 本项目的需求 | ✅ 只需服务器推送，不需要客户端发送 | ❌ 过度设计 |

SSE 是"服务器向客户端推送事件流"的最简方案。本项目的前端只接收不发送（业务数据通过 REST API），SSE 是最佳匹配。

#### 4.4.2 SSE 连接设计

```
后端 (Python / FastAPI)                        前端 (Vue 3 / TypeScript)
────────────────────────                      ──────────────────────────

EventBus                                       EventSource
   │                                              │
   ▼                                              │
_event_bus_forwarder()                            │
   │                                              │
   ├─ 推入每个订阅者队列                            │
   ├─ 写入重放缓冲区 (最近100条)                     │
   └─ 记录每个引擎最新结果                           │
   │                                              │
   ▼                                              │
_event_generator()                                │
   │                                              │
   ├─ ① 发送 connected 事件                       │──► 确认连接成功
   ├─ ② 重放缓冲区中的事件                         │──► 追上最新进度
   │     (新连接/重连能立即看到                     │
   │      当前状态，不用等待)                        │
   ├─ ③ 持续从队列拉取新事件                        │──► 实时更新 UI
   └─ ④ 每 15s 发送心跳 (keepalive)               │──► 保持连接存活
```

**重放缓冲区的设计**：

SSE 连接可能因为网络波动断开。如果没有重放机制，重连后的客户端会"错过"断开期间的事件，导致前端展示的状态滞后。

重放缓冲区有两个组成部分：
- **环形缓冲区**（最近 100 条 `ENGINE_PROGRESS` 和 `ENGINE_RESULT`）
- **最新结果字典**（每个引擎保留最新的结果，去重且覆盖）

当客户端连接后，先收到重放事件，再开始接收新事件。这意味着：
- 刚打开页面的用户能立即看到"当前进行到哪了"
- 断线重连的用户不会丢失进度

### 4.5 通信架构总览

将两条通信线路放在一起看：

```
  ┌─────────────────────────────────────────────────────────────┐
  │                        EventBus                              │
  │                                                             │
  │  线路一：引擎间通信                   线路二：前后端通信       │
  │  ────────────────                   ────────────────         │
  │                                                             │
  │  SUMMARY_READY                     ENGINE_PROGRESS          │
  │      │                            ENGINE_RESULT            │
  │      ▼                            ENGINE_ERROR             │
  │  ForumEngine                      CONSOLE_OUTPUT           │
  │      │                            FORUM_MESSAGE            │
  │      │                                  │                  │
  │      ▼                                  ▼                  │
  │  FORUM_MESSAGE                  events.py (SSE Router)     │
  │      │                                  │                  │
  │      ▼                                  │                  │
  │  forum_reader                           │                  │
  │  (缓存HOST发言)                          │                  │
  │      │                          /api/events/stream         │
  │      │                                  │                  │
  │      ▼                                  ▼                  │
  │  各引擎反思时                      Browser (EventSource)    │
  │  读取HOST建议                              │                │
  │                                     ┌─────┴─────┐          │
  │                                     │ searchStore│          │
  │                                     │ forumStore │          │
  │                                     │ appsStore  │          │
  │                                     └───────────┘          │
  └─────────────────────────────────────────────────────────────┘
```

**两条线路的分工**：

- **线路一**解决"Agent 之间怎么协作"——产生更好的分析质量
- **线路二**解决"用户怎么看到进度"——提供更好的使用体验

它们共享同一个 EventBus 基础设施，但事件类型、消费方式和关注点完全不同。这种"统一基础设施 + 分类事件"的设计，既保持了简洁性，又满足了不同场景的需求。
