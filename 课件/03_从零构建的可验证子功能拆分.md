# 尚舆分析平台：从零构建的子功能拆分

本表去除了前端部分，只保留后端、引擎、数据采集、报告生成、部署测试等核心功能点。

| 序号 | 子功能点 | 目标 |
| --- | --- | --- |
| 01 | 项目骨架与配置系统 | 搭建后端项目基础目录、配置加载、运行入口和测试入口。 |
| 02 | 后端 API 脚手架 | 建立 FastAPI 应用、路由分层（此时只有系统状态，search_router）、服务层结构（search_service）和基础接口。 |
| 03 | 事件总线与 SSE 推送 | 实现后端内部事件发布订阅，并提供实时事件流接口、前端能够通过SSE的方式，实时接收到后端事件。 |
| 04 | 单引擎最小研究工作流（InsightEngine） | 先跑通一个最小研究链路，只实现一个最小本地搜索工具 `search_topic_globally`，验证搜索、总结、反思、报告保存流程。前端能够实时查看到研究进度，能够拿到结果 |
| 05 | InsightEngine 本地搜索工具扩展 | 在最小搜索工具基础上，扩展热点查询、日期查询、评论查询、平台定向查询等工具，并统一复用 `DBResponse` 出参。 |
| 06 | InsightEngine 情感分析与聚类增强 | 使用情感分析模型，对研究结果进行情感分析；对大量搜索结果进行聚类去重和采样。 |
| 07 | 构建MediaEngine | 实现面向媒体报道、传播路径和公开资料分析的研究引擎。需要设计多个搜索工具。流程同样与InsightEngine相同 |
| 08 | 改造SearchService | 改造SearchService，实现InsightEngine和MediaEngine的并行调用，验证两个引擎能并行运行，且能够实时查看到研究进度，研究结束之后，能够在前端拿到两个引擎的结果 |
| 09 | 构建QueryEngine | 实现面向权威数据的查询研究引擎。流程同样与MediaEngine相同。需要添加权威域增强查询 |
| 10 | 改造SearchService，实现三引擎并行调用 | 改造SearchService，实现InsightEngine、MediaEngine、QueryEngine的并行调用，验证三个引擎能并行运行，且能够实时查看到研究进度，研究结束之后，能够在前端拿到三个引擎的结果 |
| 11 | 添加ForumEngine 协作讨论机制 | 通过事件机制接收三个研究引擎的摘要，收集到一定数量后调用 LLM 生成 HOST 发言，并写入 forum.log |
| 12 | 添加ReportEngine 结构化报告生成 | 前端：点击生成按钮，调用后端接口，得到一个报告生成任务id。后端：开始执行报告生成任务。前端轮询后端结果，获取报告状态 |
| 13 | 部署、测试与可观测性 | 完成容器化部署、核心测试命令、日志分流和基础排障能力。 |
